3. **Task [Extended] - Ultraplan implementation plan** (v0.1.0): Ran a 13-agent ultraplan workflow and wrote `docs/IMPLEMENTATION_PLAN.md`<br>
   **Result**: Orchestrated a multi-agent planning workflow (5 parallel research agents → 3 rival source-mapping designs + judge → plan synthesis → 2 adversarial critics → finalize; ~17.5 min, 1.23M subagent tokens) to turn the boilerplate into an executable plan for the frontend-only bidirectional Preview↔Editor feature. The mapping-design panel chose Approach A - `marked.lexer` block-ordinal correlation (clicked DOM block's ordinal index into `host.children` maps to the nth lexer token, robust to duplicate text), over text-anchor matching (B) and `data-source-line` injection (C); C was eliminated by verified fact - the apputils `Sanitizer` attribute whitelist in the installed core bundle strips `data-source-line` and previews render untrusted, so injection silently no-ops. Adversarial review caught three real defects now fixed in the plan: the context-menu selector `.jp-RenderedMarkdown` over-matches rendered notebook markdown cells (tightened to `.jp-MarkdownViewer .jp-RenderedMarkdown` + `IMarkdownViewerTracker` ownership check, AC #1); the line accumulator must replicate core's `getHeadingTokens` cumulative-newline arithmetic verbatim (`currentLine += token.raw.split('\n').length - 1`), not +1-per-token; and three ordinal-drift sources (math `removeMath`/`replaceMath`, sanitizer node removal, untrusted `<script>` warning-node prepend) need a runtime `domBlockList.length === expectedTokenCount` self-check that degrades to a re-derived `createHeaderId` heading fallback. Plan covers a first de-risking spike, dependency-ordered tasks, file-by-file change map, the pure DOM-free `src/mapping.ts` signatures, exact deps pinned to the 4.5.x line plus `marked ^17.0.6`, both command/context-menu registrations, a risks table, and a task→AC-criterion coverage matrix.
